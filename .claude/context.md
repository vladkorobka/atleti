# Контекст проекту — Атлеті

## Статус
**Фаза:** Планування завершено (v2), починається реалізація.
**Дата старту:** 2026-05-13

## Бізнес-контекст
- Цільова аудиторія: фітнес-тренери, тренери з реабілітації, схуднення
- Бізнес-модель: SaaS — MVP безкоштовний (Free plan)
- MVP фокус: core функціонал без монетизації платформи

## Тарифи
- **Free (MVP):** максимум 10 активних клієнтів на тренера
- **Pro (майбутнє):** необмежена кількість клієнтів
- `CoachProfile.plan: 'free' | 'pro'` — поле вже в схемі

## Ключові бізнес-правила
1. Клієнт не може самостійно знайти тренера — тільки через invite по нікнейму
2. Тренер **не може** надіслати invite клієнту, який вже має активного тренера
3. Клієнт відмовляється від тренера → тренер отримує сповіщення → статус `terminated`
4. Клієнт має тільки одного активного тренера одночасно
5. Баланс занять зараховується тренером вручну після офлайн-оплати
6. Клієнт може бронювати заняття тільки в межах свого балансу
7. Тренер і клієнт обидва можуть додавати заняття в календар
8. Тренер встановлює дедлайн скасування (`cancellationDeadlineHours`)
9. Клієнт може скасувати заняття тільки до дедлайну; після — заняття вважається проведеним
10. Тренер може скасувати або змінити статус **будь-якого** заняття в будь-який час

## Робота команди (ОБОВ'ЯЗКОВИЙ workflow для будь-яких змін коду)
Для **будь-яких** змін коду і фіч працює мультиагентна команда з живою дискусією.

**Склад команди:**
- Fullstack розробник
- Дизайнер
- Тестувальник
- Суворий критик

**Правила роботи критика:**
- Суворий критик критикує **всі** рішення кожного члена команди — розробника, дизайнера, тестувальника
- Критик проходиться по рішеннях кожного члена команди **мінімум 3 рази** (по можливості — більше)
- Член команди виконує **всі** вимоги критика і переробляє рішення, поки критик не дасть остаточну згоду
- Цикл "рішення → критика → доопрацювання" повторюється до нуля зауважень у критика

**Вивід у консоль (ОБОВ'ЯЗКОВО):**
- Бесіда між критиком і командою повинна виводитись у консоль, щоб користувач її бачив
- Бесіда команди по **всіх** змінах і фічах повинна виводитись у консоль
- Жодна дискусія не ховається — увесь обмін аргументами видно користувачу в реальному часі

Тільки після згоди критика і консенсусу команди — реалізація / merge.

## Технічні рішення та причини
- **Monorepo (Turborepo):** підготовка до React Native без рефакторингу
- **Cloudflare Stream:** upload з мобільного + стиснення + HLS + захищений download
- **Cloudflare R2:** зберігання файлів, дешевше S3
- **NextAuth v5:** Google OAuth + Credentials в одному провайдері
- **bcrypt:** хешування паролів для email/password auth
- **mongodb-memory-server:** integration тести без мокування БД
- **blur-sm/blur-lg + bg-white/60:** Glass UI без важких залежностей
- **cancellationDeadlineHours в CoachProfile:** кожен тренер встановлює свій ліміт

## Відкриті питання
- Форма реєстрації (поля, валідація) — продумати окремо
- Real-time для повідомлень (WebSocket vs polling) — вирішити при реалізації
- Розмір відеофайлу — ліміт завантаження (Cloudflare Stream default 200GB)
- ~~Що відбувається з балансом клієнта після відмови від тренера~~ → **Вирішено:** баланс обнуляється при переході до нового тренера. Питання повернення коштів вирішується офлайн між клієнтом і тренером — додаток не втручається.
