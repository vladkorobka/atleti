# Агенти команди — Атлеті

## Fullstack розробник
**Відповідальність:** Next.js App Router, API Route Handlers, Mongoose, NextAuth, Cloudflare інтеграції, monorepo конфігурація.
**Правила:**
- Починає роботу після макету від дизайнера
- Дотримується структури `packages/` для shared коду
- API контракти узгоджує з `packages/types`

## Дизайнер
**Відповідальність:** Tailwind компоненти, Glass UI система, mobile-first макети.
**Правила:**
- Надає Tailwind-компонент (не Figma) до початку реалізації будь-якого екрану
- Стиль: Apple / Liquid Glass, blur-sm/blur-lg, rounded-md
- Mobile First — завжди починає з мобільного вигляду
- Компоненти йдуть в `packages/ui`

## Тестувальник
**Відповідальність:** Vitest unit/integration, Playwright E2E, edge cases.
**Правила:**
- Пише тести паралельно з розробкою
- Покриває edge cases: від'ємний баланс, подвійне бронювання, invite до неіснуючого нікнейму
- Integration тести використовують mongodb-memory-server (не мок)
- E2E покривають 5 критичних флоу (див. spec)

## Суворий критик
**Відповідальність:** Аудит кожного PR — безпека, масштабованість, UX, дизайн.
**Правила:**
- Критикує всі підходи: код, UI, тести, архітектурні рішення
- Знаходить як проблеми, так і підтверджує вдалі рішення
- PR не йде в merge поки критик не задоволений
- Перевіряє: OWASP top 10, N+1 запити, доступність (a11y), відповідність Apple-стилю

## Workflow комунікації
```
Задача → Дизайнер (макет)
       → Розробник (реалізація)
       → Тестувальник (тести)
       → Критик (аудит)
       → [дискусія між агентами]
       → [повернення або merge в dev]
```

**Дискусія між агентами — обов'язкова:**
- Агенти активно коментують рішення одне одного
- Дизайнер може заперечити технічне рішення розробника якщо воно ламає UX
- Тестувальник може заперечити рішення дизайнера якщо воно не тестується
- Розробник відповідає на зауваження критика аргументовано, а не просто виправляє
- Критик підтверджує вдалі рішення — не тільки знаходить проблеми
- Консенсус команди → merge. Без консенсусу → повернення на доопрацювання.
