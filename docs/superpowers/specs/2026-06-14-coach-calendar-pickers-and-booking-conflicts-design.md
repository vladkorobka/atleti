# Дизайн: пікери календаря тренера + заборона подвійного бронювання

**Дата:** 2026-06-14
**Скоуп:** Коуч-календар (`apps/web/app/coach/calendar`), API занять, спільні UI-компоненти.

## Проблема
1. Вибір часу скрізь через нативний `<input type="time">` — незручно, особливо на мобільному.
2. Вибір дати в секціях «Заблокувати» і «Заняття» через нативний `<input type="date">` — незручно.
3. Скасоване заняття в коуч-календарі досі «займає» слот і показується бейджем «Скасовано»;
   при двох записах на один час (скасоване + нове) UI показує нестабільний результат.
4. `POST /api/coach/sessions` і `PATCH .../[sessionId]` **не перевіряють конфлікти** —
   тренер може створити кілька активних занять на той самий час.

## Рішення

### 1. `TimePicker` (packages/ui)
- Колесо в Apple-стилі: дві прокрутні колонки (години 00–23; хвилини 00,05,…,55).
- CSS scroll-snap, центрований підсвічений елемент, glass-стиль (`backdrop-blur`, `bg-white/60`, `rounded-md`).
- Props: `value: string ("HH:MM")`, `onChange: (v: string) => void`, `step?: number = 5`.
- Mobile-first: touch-scroll; десктоп: скрол/клік по елементу.
- **Крок хвилин: 5.**
- Застосувати скрізь, де зараз `<input type="time">`: робочий графік (старт/кінець),
  блок «Час» (старт/кінець), додавання та редагування заняття.

### 2. `DatePicker` (packages/ui)
- Поповер з місячною сіткою (логіка як у наявному `getMonthGrid`).
- Вибір дати → значення `"YYYY-MM-DD"`. Props: `value`, `onChange`, `min?`.
- Замінює `<input type="date">` у секціях «Заблокувати» (дата, відпустка від/до, повторюваність until)
  і «Заняття» (додавання/редагування).
- Великий місячний календар сторінки **не змінюється** — це окремий навігаційний елемент.

### 3. Скасоване заняття → вільний час
- У `buildDayTimeline` і в наборі `daysWithSessions` враховувати лише `status !== 'cancelled'`
  (тобто `scheduled` + `completed`).
- Скасований слот показується як «Вільно», без бейджа «Скасовано».
- Схема БД не змінюється — запис лишається для історії, але UI коуча його не показує.
- Клієнтський календар **не чіпаємо** — там історія скасувань корисна.

### 4. Заборона подвійного бронювання (крім Спліт)
Додати перевірку конфлікту в `POST /api/coach/sessions` і `PATCH .../[sessionId]`:
- Util `overlaps(aStart, aDur, bStart, bDur)` — перетин напіввідкритих інтервалів `[start, start+dur)`.
  Стик впритул (кінець == початок) — **не** конфлікт.
- Шукаємо заняття коуча зі `status: 'scheduled'`, чий інтервал перетинається з новим
  (PATCH виключає `_id` самого заняття).
- Немає перетинів → дозволено.
- Є перетин → дозволено **тільки** якщо нове `type === 'split'` **і всі** перетнуті теж `'split'`.
  Інакше — `409` з укр. повідомленням («На цей час уже заплановано заняття.
  Поверх можна додати лише Спліт-заняття.»).
- Скасовані/завершені не блокують.
- Це закриває баг п.3: після скасування слот вільний, але два активних заняття на один час неможливі.

## Тести (Vitest)
- `overlaps()`: стик впритул → false; частковий перетин → true; вкладений → true.
- `POST /api/coach/sessions`: regular поверх regular → 409; split поверх split → 201;
  split поверх regular → 409; regular поверх split → 409; перетин зі скасованим → 201.
- `PATCH`: перенос на зайнятий час → 409; перенос self на свій же час → 201 (виключення себе).
- Таймлайн коуча: скасоване заняття не займає слот (рендериться «Вільно»).

## Поза скоупом
- Клієнтський `available-slots` (exact-time matching) — не чіпаємо.
- Клієнтська історія скасувань — лишається.
- Обмеження кількості Спліт-занять на слот / перевірка різних клієнтів — майбутнє (YAGNI).
